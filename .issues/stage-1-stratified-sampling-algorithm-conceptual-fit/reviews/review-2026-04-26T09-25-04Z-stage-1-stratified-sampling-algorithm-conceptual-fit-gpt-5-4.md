<finding severity="high" id="H1">
  <title>The signed audit trail does not prevent seed-shopping or prove an impartial lottery</title>
  <location>packages/core/src/stage1/types.ts:50-85; sortition-tool/08-product-redesign.md:18-25</location>
  <description>The method is reproducible conditional on a chosen seed, but the current audit model does not show that the seed itself was chosen fairly. The audit document records `seed` and `seed_source`, where the source can be `user` or `unix-time-default`, and the signed snapshot is produced after the draw, not before it (`packages/core/src/stage1/types.ts:50-85`; `sortition-tool/08-product-redesign.md:18-25`). That means an operator can rerun the lottery across many seeds and publish only the preferred result while still producing a valid signed JSON. For sortition legitimacy, that is a material gap: recent work treats transparency and manipulation-robustness as distinct design goals, not as automatic by-products of reproducibility (Baharav &amp; Flanigan, “Fair, Manipulation-Robust, and Transparent Sortition”, arXiv:2406.15009, abstract lines 42-46, https://arxiv.org/abs/2406.15009).</description>
  <fix>Keep deterministic sampling, but require the seed to come from a publicly auditable entropy source and commit to it before selection: for example a 128-bit seed generated with Web Crypto (`Crypto.getRandomValues()` is the browser-standard cryptographically strong source: https://developer.mozilla.org/en-US/docs/Web/API/Crypto/getRandomValues) or a public ceremony / commit-reveal scheme. The signed audit should include a pre-draw commitment, the revealed seed, and an algorithm version so the signature proves not just consistency, but fairness of the draw procedure.</fix>
</finding>

<finding severity="high" id="H2">
  <title>The signed audit document does not bind the signature to the actual selected invitation list</title>
  <location>packages/core/src/stage1/types.ts:54-85</location>
  <description>`Stage1AuditDoc` signs the input hash, axes, pool/target sizes, per-stratum counts, warnings, and metadata, but it does not include `selected` row indices, stable person IDs, or any equivalent selected-list digest (`packages/core/src/stage1/types.ts:54-85`). As a result, a verifier can confirm the allocation table, but cannot confirm that the exported CSV is exactly the one produced by the audited run unless they independently rerun the algorithm against the private source CSV. That is weaker than the claimed downstream verification model in which “anyone re-runs and confirms,” especially because the source CSV is a protected resident-register extract rather than a public dataset (`sortition-tool/08-product-redesign.md:22-25`; `research/03-legal-framework-and-best-practices.md:323-338`).</description>
  <fix>Include one of: selected row indices in deterministic order, stable selected person IDs, or at minimum a signed digest of the selected output CSV plus the exact algorithm/version metadata. If privacy concerns rule out publishing identities broadly, keep the signed selected-list binding in the internal audit artifact available to the municipality and auditor.</fix>
</finding>

<finding severity="high" id="H3">
  <title>The determinism contract is not robust for actual German/Austrian text data</title>
  <location>docs/stage1-algorithm.md:35-39,135-139; packages/core/src/stage1/stratify.ts:76-84,136-145; scripts/stage1_reference.py:140-152</location>
  <description>The spec explicitly warns that TypeScript uses `localeCompare()` while Python uses codepoint sorting, and that umlauts can therefore produce different stratum orders (`docs/stage1-algorithm.md:35-39`). The production code still sorts with `localeCompare()` in TypeScript and `sorted()` in Python (`packages/core/src/stage1/stratify.ts:76-84,136-145`; `scripts/stage1_reference.py:140-152`). For a tool aimed at German and Austrian municipalities, where district and locality names regularly contain non-ASCII characters, this is not a theoretical corner case; it undermines byte-identical cross-runtime reproduction and therefore weakens the audit story on real data rather than only fixtures.</description>
  <fix>Define a language-independent canonical comparator in the specification and implementations, such as bytewise UTF-8 order over normalized strings, and record the normalization/comparator rule in the audit metadata. “Transliterate the real data first” is not a sufficient methodological contract for a public-lottery tool targeted at DE/AT administrative data.</fix>
</finding>

<finding severity="medium" id="M1">
  <title>The no-redistribution underfill policy is effectively unreachable in the current design</title>
  <location>packages/core/src/stage1/stratify.ts:28-50,63-90,128-157; docs/stage1-algorithm.md:83-84,135-139</location>
  <description>The review prompt asks whether a stratum allocated 3 can contain only 2 people, but under the current algorithm that situation does not arise for occupied strata. `bucketize()` creates strata only for observed cells (`packages/core/src/stage1/stratify.ts:28-50`), `targetN` is constrained to be at most the pool size (`packages/core/src/stage1/stratify.ts:128-132`), and Hamilton allocates each occupied stratum either `floor(q_h)` or `ceil(q_h)` where `q_h = targetN * N_h / N_total &lt;= N_h` (`packages/core/src/stage1/stratify.ts:63-90`). Therefore `n_h_target &gt; N_h` cannot occur for any occupied stratum, so the documented underfill/redistribution debate is presently moot (`docs/stage1-algorithm.md:135-139`).</description>
  <fix>Document this explicitly. If future product requirements add explicit empty cells, minimum-per-cell rules, oversampling, or multi-wave exclusions, then underfill becomes real and redistribution policy must be re-evaluated. Today, the current “warn and do not redistribute” rule is not an operational feature of the implemented proportional design.</fix>
</finding>

<finding severity="medium" id="M2">
  <title>The validation report overstates what the statistical test proves</title>
  <location>docs/stage1-validation-report.md:7-9,34-45,56-60; scripts/stage1_reference.py:4-14; scripts/stage1_statistical_test.py:22-25,65-83,95-157</location>
  <description>The Python “reference” is explicitly designed to mirror the TypeScript implementation bit-for-bit, so the cross-runtime check is a conformity test, not an independent methodological validation (`scripts/stage1_reference.py:4-14`). The statistical test then reuses the same implementation, derives each person’s target probability from that implementation’s own allocation table, and reports exact stratum-level agreement that is guaranteed by construction because Hamilton allocation is deterministic (`scripts/stage1_statistical_test.py:65-83,134-153`; `docs/stage1-validation-report.md:34-45`). The per-person test is still useful as a conditional check that within-stratum shuffling behaves like SRSWOR, but it does not establish that the overall method is unbiased in any broader sense, nor does it rule out shared design errors, which the report itself partly concedes (`docs/stage1-validation-report.md:56-60`).</description>
  <fix>Reframe the claim as: “the implementation appears to produce uniform within-stratum draws conditional on the chosen proportional allocation.” For stronger validation, compare allocations and selections with an external survey-sampling implementation such as SAS PROC SURVEYSELECT (`ALLOC=PROP`, `method=srswor`) or R’s `sampling::strata` (`method="srswor"`): https://support.sas.com/documentation/cdl/en/statug/68162/HTML/default/statug_surveyselect_details19.htm and https://rdrr.io/cran/sampling/src/R/strata.r.</fix>
</finding>

<finding severity="low" id="L1">
  <title>The Hamilton justification is acceptable, but the claim that it is the DACH standard is unsupported</title>
  <location>docs/stage1-algorithm.md:77</location>
  <description>The specification states that Hamilton is “die DACH-Standard-Praxis in der amtlichen Statistik,” but the repository does not cite a source for that claim (`docs/stage1-algorithm.md:77`). Established survey tools do support proportional allocation with integer rounding, but the cited public documentation here is generic survey-sampling documentation rather than a DACH-specific statistical standard; for example SAS documents proportional allocation and redistribution when a stratum cap binds, without framing Hamilton as a normative public-sector standard (https://support.sas.com/documentation/cdl/en/statug/68162/HTML/default/statug_surveyselect_details19.htm).</description>
  <fix>If this claim matters for external justification, replace it with a sourced and narrower statement: Hamilton/largest-remainder is a defensible quota-respecting rounding rule for proportional stratified sampling, but it is one of several acceptable integer-allocation conventions.</fix>

<answer id="1" verdict="defensible">Yes for this use case. OECD describes the common recruitment pattern for representative deliberative processes as a two-stage process in which a large first draw is made and then a second stratified draw forms the final panel; the first-stage variables are typically age, gender, geographic locality, and one socio-demographic indicator (OECD 2020, PDF pp. 92-93, lines 3738-3760, https://www.oecd.org/content/dam/oecd/en/publications/reports/2020/06/innovative-citizen-participation-and-new-democratic-institutions_11aa2baf/339306da-en.pdf). The repo’s own product and legal docs match that framing: Stage 1 is a simple O(N) invitation draw from Melderegister fields, while stage 3 is where the hard fairness optimization belongs (`sortition-tool/07-two-stage-workflow-analysis.md:12-18,48-57,66-80`; `research/03-legal-framework-and-best-practices.md:208-246`). Systematic, cluster, multi-stage, or cube-method designs are not meaningfully better for a 6,000-to-300 municipal invitation list with only a few legally available axes; the dominant representativeness problem is nonresponse, which is addressed downstream by the second-stage sortition process, not by making stage 1 more exotic (OECD PDF pp. 92-93 lines 3738-3760; Nature 2021 lines 91-119, https://www.nature.com/articles/s41586-021-03788-6).</answer>
<answer id="2" verdict="defensible">Hamilton is defensible and the choice is second-order here. The implemented rule preserves the total exactly and gives each occupied cell either its lower or upper quota (`packages/core/src/stage1/stratify.ts:63-90`). In a 300-of-6,000 invitation draw, the concrete effect of switching to Webster/Sainte-Laguë or d’Hondt is usually a one-invite difference in a few boundary cells, whereas the larger methodological risks in this workflow are response-rate distortion and second-stage feasibility (`research/03-legal-framework-and-best-practices.md:218-244`; `sortition-tool/07-two-stage-workflow-analysis.md:52-57,61-69`). I do not see a case for preferring a divisor method on representativeness grounds for stage 1 alone. The repo should, however, drop the unsupported claim that Hamilton is the DACH standard (`docs/stage1-algorithm.md:77`).</answer>
<answer id="3" verdict="defensible">Proportional allocation is the right default because stage 1 is not an estimation exercise with a known study variable whose variance should drive sample allocation. Neyman allocation is designed to minimize estimator variance, and standard survey software presents it in that estimation context, alongside proportional and optimal allocation (SAS PROC SURVEYSELECT docs: https://support.sas.com/documentation/cdl/en/statug/63962/HTML/default/statug_surveyselect_a0000000206.htm). Here the goal is descriptive invitation coverage on the legally available frame variables, after which stage 3 corrects volunteer-pool imbalance using quota-aware panel selection (`research/03-legal-framework-and-best-practices.md:213-246`; Nature 2021 lines 91-119). I do not see a missing methodological argument for Neyman unless the product is repurposed as a survey-estimation tool.</answer>
<answer id="4" verdict="questionable">Mulberry32 is acceptable for deterministic replay, but questionable as the randomness protocol for a public lottery. The implementation is a 32-bit deterministic PRNG seeded from a single numeric seed (`packages/core/src/pool/mulberry32.ts:1-24`; `packages/core/src/stage1/types.ts:19-20,57-58`). Once the seed is published in the audit, a CSPRNG would not make the replayed stream “more secret”; however, the current design leaves the seed under user control and signs the audit after the run, which enables seed-shopping (`packages/core/src/stage1/types.ts:50-85`; `sortition-tool/08-product-redesign.md:18-25`). For legitimacy, the better change is not merely “swap Mulberry32 for a CSPRNG,” but “generate or reveal the seed through a publicly auditable process.” If the product wants browser-native strong entropy, Web Crypto provides `getRandomValues()` (https://developer.mozilla.org/en-US/docs/Web/API/Crypto/getRandomValues).</answer>
<answer id="5" verdict="questionable">As implemented, this policy is mostly moot. Because the code only allocates occupied strata and enforces `targetN &lt;= rows.length`, Hamilton cannot allocate more invitations to an occupied stratum than that stratum contains (`packages/core/src/stage1/stratify.ts:28-50,63-90,128-157`). So the current no-redistribution rule is not really making a live methodological tradeoff today. If future versions add minimum-per-cell rules or explicit empty cells, then a non-redistribution policy would become questionable because standard survey software redistributes residual sample size when a stratum cap binds (SAS docs lines 31-34 at https://support.sas.com/documentation/cdl/en/statug/68162/HTML/default/statug_surveyselect_details19.htm). Today I would document the policy as dormant rather than defend it as a substantive design choice.</answer>
<answer id="6" verdict="defensible">Shared-RNG plus canonical stratum order is conceptually acceptable for deterministic SRSWOR within each stratum; it does not introduce bias by itself because the within-stratum shuffle is still Fisher-Yates and each stratum draw is uniform conditional on its allocation (`packages/core/src/stage1/stratify.ts:93-101,144-171`). A per-stratum RNG derived from `(master_seed, stratum_key)` would be cleaner because it removes cross-stratum coupling and makes the “empty strata do not advance RNG state” convention unnecessary, but I would treat that as an auditability refinement rather than a methodological correction. The more important issue is that the claimed canonical order is not stable on non-ASCII data (`docs/stage1-algorithm.md:35-39`; `scripts/stage1_reference.py:140-152`).</answer>
<answer id="7" verdict="defensible">The absence of IPF/raking is not material for this stage. OECD’s two-stage guidance puts richer quota management on the second-stage draw from respondents, not on the first-stage invitation sample, and the legally available stage-1 fields are limited (`research/03-legal-framework-and-best-practices.md:208-246`). The current design already works on the full cross-product of the selected axes, which is stronger than only matching one marginal at a time (`packages/core/src/stage1/stratify.ts:13-18,28-50`). IPF becomes relevant when one wants simultaneous marginal targets without enumerating the full cross-product, or when final-panel quotas include variables unavailable in the register; that is a stage-3 concern, not a stage-1 blocker (Nature 2021 lines 91-119; `research/05-sortition-algorithm.md:101-187`).</answer>
<answer id="8" verdict="questionable">The validation evidence is useful but overclaimed. The per-person frequency test is a reasonable check that the implemented Fisher-Yates draw behaves like SRSWOR conditional on the fixed `n_h_target` values, because all individuals in a stratum should have probability `n_h_target / N_h` (`scripts/stage1_statistical_test.py:65-132`). But the test is not an independent validation of the overall method, because it reuses the same implementation to obtain the “theoretical” allocation table, and the exact stratum-level totals are tautological once allocation is deterministic (`scripts/stage1_statistical_test.py:69-83,134-153`; `docs/stage1-validation-report.md:43-45`). So the answer to the circularity concern is: partly yes for the allocation claims, no for the narrower within-stratum uniformity check.</answer>

<verdict value="warn" critical="0" high="3" medium="2">
  <summary>The core algorithm choice is mostly right for this product: proportional stratified SRSWOR is an appropriate stage-1 method for municipal citizen-assembly invitation lists constrained to a few Melderegister variables, and there is no clearly better alternative that would materially improve representativeness before the second-stage respondent selection. The significant concerns are elsewhere: the current audit design does not yet prove an impartial lottery, does not bind the signature to the actual selected list, and is not robust across real German/Austrian text data because of the documented sorting divergence. Those are meaningful methodological/auditability issues, but they do not make the underlying stage-1 sampling method itself conceptually wrong for the use case.</summary>
  <blockers>
    <blocker>No blocking statistical flaw in the choice of proportional stratified sampling itself was found.</blocker>
    <blocker>Before claiming strong public verifiability, the audit protocol needs seed-governance hardening and signed binding to the actual selected list.</blocker>
