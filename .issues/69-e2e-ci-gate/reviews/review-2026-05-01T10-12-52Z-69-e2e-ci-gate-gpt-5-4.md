## 2. Cache-key correctness for Playwright browsers

### Medium — The cache is shared across both matrix legs, but each leg installs a different browser, so the cache can permanently contain only one browser

**Evidence:** `.github/workflows/deploy.yml:99-111` uses one cache key for both matrix jobs:

- `key: playwright-${{ runner.os }}-${{ hashFiles('pnpm-lock.yaml') }}`
- `path: ~/.cache/ms-playwright`
- each job then runs `playwright install --with-deps ${{ matrix.browser }}`

With `actions/cache`, the cache is scoped by key. On the first cold run, the chromium leg and firefox leg both restore an empty cache, then each populates `~/.cache/ms-playwright` with only its own browser. Only one save wins for that shared key, so the stored cache ends up with whichever browser saved first. On later runs, an exact-key restore prevents upgrading that cache to include the missing browser, which means one matrix leg will keep re-downloading its browser binaries on most runs.

**Why it matters:** This does not break correctness, but it defeats most of the intended CI speedup for one browser leg and makes the cache behavior misleading.

**Action:** Split the cache by browser, for example:

```yaml
key: playwright-${{ runner.os }}-${{ matrix.browser }}-${{ hashFiles('apps/web/package.json') }}
restore-keys: |
  playwright-${{ runner.os }}-${{ matrix.browser }}-
```

Alternatively, install both browsers before the cache is saved, but that removes the current per-browser download optimization.

### Low — `pnpm-lock.yaml` is broader than necessary for browser-version invalidation

**Evidence:** The cache key in `.github/workflows/deploy.yml:104` hashes `pnpm-lock.yaml`, while `@playwright/test` is pinned directly in `apps/web/package.json:22`.

**Why it matters:** Any unrelated workspace dependency change invalidates the browser cache even when the Playwright version did not change. That produces avoidable churn. `apps/web/package.json` is the narrower invalidation surface and still catches explicit Playwright bumps in this repo because the version is pinned there.

**Action:** Prefer `hashFiles('apps/web/package.json')` unless the team explicitly wants browser cache invalidation on every lockfile movement.

## 5. Subtle breakage risk to existing build/deploy/smoke jobs

### Medium — The new failure-artifact steps promise traces and an HTML report, but the checked-in Playwright config does not generate either

**Evidence:**

- `.github/workflows/deploy.yml:129-142` uploads `apps/web/test-results/` as “traces” and `apps/web/playwright-report/` as an “HTML report”.
- `apps/web/playwright.config.ts:9-12` sets `reporter: 'list'` and does not set `use.trace`.
- By contrast, `apps/web/playwright-live.config.ts:16-20` explicitly enables `trace: 'retain-on-failure'` for the smoke workflow, which highlights the missing setting in the default config.

**Why it matters:** On e2e failure, both new upload steps are likely to publish nothing. `if-no-files-found: ignore` hides that problem, so the workflow will look healthy while dropping the main debugging artifacts the issue explicitly set out to provide.

**Exact fix:** Make the e2e run actually produce those artifacts. For example:

```ts
// apps/web/playwright.config.ts
use: {
  baseURL: 'http://127.0.0.1:4173',
  headless: true,
  trace: 'retain-on-failure',
},
reporter: process.env.CI ? [['list'], ['html', { open: 'never' }]] : 'list',
```

Or keep the config unchanged and add `--trace=retain-on-failure --reporter=list,html` to the CI `playwright test` command.

## 6. Playwright config correctness

### Low — `testIgnore` also blocks explicit-path runs, so the documented manual override no longer exists

**Evidence:**

- `apps/web/playwright.config.ts:4-6` adds `testIgnore: ['**/_*.spec.ts']`.
- Playwright matches `testIgnore` against absolute file paths, so this pattern excludes `_visual-iteration*.spec.ts` regardless of whether the file is discovered normally or passed explicitly.
- The branch plan claims the opposite at `.issues/69-e2e-ci-gate/PLAN.md:59-79`.
- Local verification from this worktree: `pnpm exec playwright test /root/workspace/.worktrees/69-e2e-ci-gate/apps/web/tests/e2e/_visual-iteration-65.spec.ts --list` returns `No tests found`.

**Why it matters:** CI gating is correct, but the documented “run the visual generator by explicit path” escape hatch is false. That makes screenshot regeneration more awkward than the plan suggests.

**Action:** If those visual specs need to stay runnable, move them out of `tests/e2e/` or create a second Playwright config for visual generation instead of relying on explicit-path invocation.
